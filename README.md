# 0320-project

## 팀장 조윤주 오류 부분

1. 개발 환경 설정 오류 및 해결

초기 개발 과정에서 main.py 실행 테스트 중, .venv의 파일 경로를 혼동하는 문제가 발생하였다.
이로 인해 FastAPI 서버 실행이 정상적으로 이루어지지 않았으며, 원인을 파악하는 데 시간이 소요되었다.

해당 문제는 강사님의 피드백을 통해 경로 설정을 수정함으로써 해결하였다.
이후에는 별도로 가상환경에 직접 진입하지 않더라도, uv run fastapi run 명령어를 통해 Python 3.10 환경을 정상적으로 연결하고 애플리케이션을 실행할 수 있음을 확인하였다.

본 사례를 통해, 개발 환경 실행 방식에 대한 이해가 부족했음을 인지하였으며,
특히 “가상환경은 반드시 직접 활성화해야 한다”는 고정관념이 문제의 원인으로 작용했음을 확인하였다.

-------------------------------------

2. Spark 데이터 처리 흐름에 대한 이해 오류

데이터 처리 과정에서 전체적인 흐름에 대한 이해 부족으로 시행착오가 발생하였다.

초기에는 MariaDB 또는 CSV 원본 데이터를 spark.read.csv()로 불러온 뒤,
이를 createOrReplaceTempView()를 활용하여 View 형태로 변환하고,
이후 spark.sql("""쿼리""") 방식으로 데이터를 처리해야 한다고 판단하였다.

그러나 실제로는 반드시 SQL 기반 처리 방식이 필요한 것은 아니었으며,
Spark DataFrame API를 활용한 방식으로도 충분히 데이터 가공이 가능함을 확인하였다.

최종적으로는 아래와 같은 방식으로 DataFrame 연산을 통해 데이터를 처리하였다.

- 컬럼 정제 (trim, 형 변환)
- 불필요한 문자 제거 (regexp_replace)
- 그룹화 및 집계 (groupBy, agg)
- 정렬 (orderBy)

이를 통해 불필요하게 SQL View를 생성하지 않고도, 보다 직관적이고 효율적인 데이터 처리 흐름을 구축할 수 있었다.

본 경험을 통해, Spark에서 SQL 방식과 DataFrame API 방식의 차이를 명확히 이해하는 것이 중요하다는 점을 학습하였다.

-------------------------------------

3. 데이터 품질 이슈 및 원인 분석

데이터 파이프라인을
(View → DataFrame → Pandas → JSON → API 응답 → React)
까지 완성한 이후, 결과 데이터를 검증하는 과정에서 이상 현상이 발견되었다.

구체적으로, 날짜 형식이 YYYY-MM-DD가 아닌 DD/MM/YYYY 형태로 일부 데이터에서 나타나는 문제가 발생하였다.

초기에는 코드 로직 또는 시스템 처리 과정의 오류로 판단하였으나,
원인 분석 결과, CSV 원본 데이터 변환 과정에서 날짜 포맷이 비일관적으로 변경된 것이 문제의 근본 원인이었다.

즉, 일부 데이터가 다음과 같이 서로 다른 형식으로 혼재되어 있었다.

- 2017-01-01 (정상 형식)
- 05/12/2017 (비일관 형식)

본 사례를 통해 다음과 같은 중요한 교훈을 도출하였다.

- 데이터 이상 발생 시, 코드뿐만 아니라 원본 데이터의 품질을 우선적으로 검증해야 한다.
- 데이터 전처리 과정에서의 작은 변경이 전체 시스템 결과에 큰 영향을 미칠 수 있다.
- 문제 발생 시 “코드 오류”에만 집중하기보다, 데이터 자체를 의심하는 접근이 필요하다.

-------------------------------------

4. 데이터 처리 구조 설계 오류 및 개선

초기 설계 단계에서는 Apache Spark를 활용하여 데이터 연산을 사전에 수행하고, FastAPI에서는 단순 조회만 수행하는 구조를 목표로 하였다.

그러나 실제 구현 과정에서는 성능 및 개발 시간 제약으로 인해, 사전 계산된 데이터를 DB에 직접 적재하고 이를 조회하는 방식으로 변경하게 되었다.

이 과정에서 이론적으로 이상적인 구조와 실제 프로젝트 환경에서의 효율적인 구조를 명확히 구분하지 못해, 불필요한 시행착오와 시간 소모가 발생하였다.

본 사례를 통해 다음과 같은 점을 학습하였다.

데이터 처리 구조는 이론보다 “성능과 응답속도” 기준으로 판단해야 한다.

실시간 처리(Spark)와 사전 집계(DB)의 역할을 명확히 분리하는 것이 중요하다.

초기 설계 단계에서 데이터 규모를 고려한 현실적인 아키텍처 선택이 필요하다.

-------------------------------------

5. 컬럼 설계 오류 (구분 컬럼 혼합)

유동인구 통계 테이블 생성 과정에서 “구분” 컬럼에 승차/하차 정보와 공휴일 정보가 동시에 저장되는 문제가 발생하였다.

이로 인해 하나의 컬럼이 서로 다른 의미를 가지게 되었고, 데이터 정합성이 크게 훼손되었다.

해당 문제는 강사님의 피드백을 통해 조기에 발견되었으며, 이후 “승하차 구분”과 “공휴일 구분” 컬럼을 분리하는 방식으로 수정하였다.

본 사례를 통해 다음과 같은 교훈을 도출하였다.

컬럼은 반드시 하나의 의미(단일 책임)를 가져야 한다.

데이터 모델링 단계에서의 작은 실수가 전체 분석 결과에 큰 영향을 미칠 수 있다.

스키마 설계는 구현 이전에 충분히 검증되어야 한다.

-------------------------------------

6. NULL 및 0 값 처리 기준 미정립

데이터 전처리 과정에서 NULL 값과 0 값에 대한 명확한 기준 없이 처리함으로써, 동일 연도 내 역 개수가 일관되지 않게 나타나는 문제가 발생하였다.

이는 집계 결과의 왜곡뿐만 아니라, 분석 결과의 신뢰도에도 영향을 미쳤다.

본 경험을 통해 다음과 같은 기준을 정립하였다.

NULL: 데이터가 존재하지 않는 경우 → 분석 시 제외 또는 보정 필요

0: 실제 값 → 그대로 유지

또한, 데이터 전처리 단계에서 이러한 기준을 사전에 정의하는 것이 중요하다는 점을 확인하였다.

-------------------------------------

7. 데이터 타입 및 쿼리 최적화 이슈

초기 테이블 생성 시 날짜 데이터를 TEXT 또는 INT 타입으로 저장하여, 조회 및 정렬 성능 저하 문제가 발생하였다.

이후 날짜 컬럼을 DATE 타입으로 변경하면서, 쿼리 성능이 유의미하게 개선되는 것을 확인하였다.

또한, Subquery 중심으로 작성된 쿼리는 성능 측면에서 비효율적이었으며, 이를 JOIN 기반 구조로 개선하였다.

다만, JOIN 사용 시 중복 데이터 발생 가능성이 존재하기 때문에, 상황에 따라 LEFT JOIN을 활용하는 방식으로 안정성을 확보하였다.

본 사례를 통해 다음과 같은 점을 학습하였다.

데이터 타입 설계는 성능에 직접적인 영향을 미친다.

Subquery보다 JOIN이 일반적으로 성능상 유리하다.

JOIN 사용 시 데이터 중복 및 정합성 문제를 반드시 고려해야 한다.

-------------------------------------

개발 환경 이해, 데이터 처리 방식 선택, 그리고 데이터 품질 검증의 중요성을 함께 학습할 수 있었습니다.”

## 팀원 이채훈 오류 부분

1. 컬럼 인덱스 지정 오류

데이터 전처리 과정에서 특정 컬럼을 참조하는 과정에서 오류가 발생하였다.
원인은 실제 데이터 구조를 정확히 확인하지 않은 상태에서, 0번 인덱스를 기준으로 컬럼을 탐색한 데 있었다.

해당 데이터는 다수의 공백 컬럼(“, , , …”) 이후 실제 컬럼(예: 인원명)이 배치된 구조였으며,
이로 인해 잘못된 컬럼을 참조하게 되어 오류가 발생하였다.

본 사례를 통해, 데이터 로딩 이후 컬럼 구조 및 스키마를 사전에 검증하는 과정의 중요성을 확인하였다.

-------------------------------------

2. 날짜 형식 불일치로 인한 적재 오류

데이터 정제 및 적재 과정에서 날짜 형식 불일치로 인한 오류가 발생하였다.

특히 특정 연도(예: 2020년)의 데이터가 전체적으로 YYYY/MM/DD 형식으로 저장되어 있었으며,
다른 연도 데이터와 형식이 일치하지 않아 처리 과정에서 오류가 발생하였다.

이는 일부 데이터만 부분적으로 형식이 다른 것이 아니라,
특정 기간 데이터 전체가 상이한 포맷을 가지는 경우라는 점에서 추가적인 주의가 필요하였다.

본 사례를 통해, 데이터 전처리 시 기간별/배치별 데이터 일관성 검증이 필수적임을 확인하였다.

-------------------------------------

3. Python 버전 불일치 문제

개발 환경과 서버 환경 간 Python 버전 차이로 인해 실행 오류가 발생하였다.

개발 환경에서는 Python 3.13.x 버전을 사용하고 있었으나,
서버 환경은 Python 3.10.x 버전을 사용하고 있어 일부 코드 및 패키지 호환성 문제가 발생하였다.

해당 문제는 서버 환경에 맞춰 Python 버전을 조정함으로써 해결하였다.

본 사례를 통해, 개발 환경과 배포 환경 간 버전 일치(환경 표준화)의 중요성을 확인하였다.

-------------------------------------

4. 함수 호출 오류 (오타)

함수 실행 과정에서 오류가 발생하였으며,
원인 분석 결과 함수명 또는 변수명에 대한 단순 오타로 확인되었다.

코드 로직 자체에는 문제가 없었으나,
사소한 오타로 인해 실행이 중단되는 문제가 발생하였다.

본 사례를 통해, 기본적인 코드 작성 시에도
정확한 명명 규칙 및 코드 검증의 중요성을 다시 한번 확인하였다.

-------------------------------------

5. 대용량 데이터 처리 방식 개선

데이터 로딩 과정에서 전체 데이터를 한 번에 불러오는 방식으로 인해
성능 저하 및 처리 지연 문제가 발생하였다.

이를 해결하기 위해 데이터를 청크(Chunk) 단위로 나누어 로드 및 처리하는 방식으로 개선하였다.

이 방식은 메모리 사용을 효율적으로 관리할 수 있으며,
대용량 데이터 처리 시 안정성을 확보하는 데 효과적이었다.

본 사례를 통해, 데이터 처리 시 데이터 규모를 고려한 처리 전략 수립의 필요성을 확인하였다.

-------------------------------------

6. 중복 컬럼명으로 인한 데이터 처리 오류

데이터셋 내 동일한 이름의 컬럼(“구분”)이 복수로 존재하여 오류가 발생하였다.

문제는 동일한 컬럼명임에도 불구하고,
각 컬럼이 서로 다른 의미의 데이터를 포함하고 있었다는 점이다.

일부 “구분” 컬럼: 승차/하차 정보 포함

다른 “구분” 컬럼: 별도의 분류 데이터 포함

이로 인해 잘못된 컬럼을 참조하는 문제가 발생하였으며,
최종적으로는 승/하차 정보가 포함된 컬럼을 기준으로 재선별하여 처리하였다.

본 사례를 통해, 데이터 처리 시
컬럼명뿐만 아니라 실제 데이터 의미(semantic)를 기준으로 검증하는 과정의 중요성을 확인하였다.

-------------------------------------

8. 팀 내 데이터 적재 오류 및 영향

팀원 작업 과정에서, HeidiSQL 환경에서 사전 정의되지 않은 함수를 사용하여 쿼리를 실행하는 문제가 발생하였다.

이로 인해 2019년 및 2020년 데이터가 승차/하차 테이블에 정상적으로 적재되지 않았으며,
해당 테이블을 기반으로 생성된 모든 파생 테이블 및 View에 연쇄적인 데이터 오류가 발생하였다.

특히, 코로나 전후 비교 분석을 수행하는 데 있어 핵심 데이터가 누락되면서, 분석 결과의 신뢰성이 크게 저하될 수 있는 상황이었다.

해당 문제는 이후 데이터 검증 과정에서 발견되었으며, 재적재를 통해 정상화하였다.

본 사례를 통해 다음과 같은 개선 방향을 도출하였다.

쿼리 실행 전 함수 및 객체 존재 여부를 반드시 확인해야 한다.

핵심 테이블 적재 이후에는 Row Count, 기간 분포 등 검증 절차를 필수적으로 수행해야 한다.

데이터 파이프라인에서는 “한 단계의 오류가 전체 시스템에 영향을 미친다”는 점을 고려해야 한다.

-------------------------------------

“팀 전체적으로 다양한 오류를 경험하면서,
단순 구현을 넘어 데이터 구조 이해, 환경 일관성, 그리고 데이터 품질 검증이
프로젝트 성공에 핵심 요소임을 확인할 수 있었습니다.”

-------------------------------------
