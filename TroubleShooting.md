# TroubleShooting.md

## 1. 개발 환경 설정 오류
- 문제
FastAPI 실행 시 가상환경(.venv) 경로 혼동으로 인한 서버 실행 실패

- 원인
가상환경을 매번 수동으로 활성화해야 한다는 도구 활용 이해도 부족

- 해결
uv run fastapi run 명령어를 통해 환경 자동 연결 및 실행 방식 최적화

- 배운 점
개발 도구별 환경 실행 메커니즘을 정확히 파악하는 것이 환경 구성의 핵심임

## 2. Spark 처리 방식 선택 오류
- 문제
SQL View 기반 처리 방식만 고집하여 복잡한 로직 구현 시 비효율 발생

- 원인
Spark DataFrame API의 기능 및 활용법에 대한 학습 부족

- 해결
DataFrame API(groupBy, agg 등) 기반으로 전환하여 로직 간소화 및 가독성 확보

- 배운 점
데이터 프레임워크는 상황에 따라 SQL과 API 방식 중 더 직관적인 도구를 선택해야 함

## 3. 데이터 품질 이슈 (날짜 포맷 불일치)
- 문제
날짜 형식이 혼재(YYYY-MM-DD vs DD/MM/YYYY)되어 데이터 분석 시 오류 발생

- 원인
데이터 소스 변환 및 병합 과정에서 포맷 통일 규격 미정립

- 해결
데이터 전처리(Raw-Int) 단계에서 포맷 표준화 로직 적용

- 배운 점
코드 로직 수정 이전에 원본 데이터의 규격과 품질 검증이 최우선되어야 함

## 4. 데이터 처리 구조 설계 오류
- 문제
실시간 처리 구조 설계로 인해 대용량 데이터 처리 시 성능 및 응답 시간 저하

- 원인
데이터 규모와 비즈니스 요구사항을 고려하지 않은 과도한 아키텍처 설계

- 해결
사전 집계 후 DB에 적재하는 배치(Batch) 기반 구조로 변경하여 성능 최적화

- 배운 점
시스템 설계는 이론적 이상향보다 실제 데이터 규모와 처리 성능을 우선시해야 함

## 5. 컬럼 모델링 설계 오류
- 문제
하나의 컬럼에 복수의 의미가 혼합되어 데이터 추출 시 불필요한 가공 발생

- 원인
데이터 모델링 단계에서 단일 책임 원칙(SRP)에 대한 고려 부족

- 해결
컬럼 분리 및 스키마 재설계를 통해 데이터 원자성 확보

- 배운 점
데이터 스키마 설계는 초기 단계에서 향후 확장성을 고려해 철저히 검증해야 함

## 6. NULL 및 0 처리 기준 부재
- 문제
수치형 데이터 집계 시 실제 0과 데이터 누락(NULL)이 혼재되어 결과 왜곡 발생

- 원인
데이터 도메인 정의 및 결측치 처리 기준 미정립

- 해결
비즈니스 로직에 따른 NULL과 0의 처리 기준 명확화 및 전처리 적용

- 배운 점
데이터 정의 단계에서 결측치 처리 정책을 수립하는 것이 정합성 확보의 기초임

## 7. 데이터 타입 및 쿼리 최적화 실패
- 문제
날짜 데이터를 TEXT 타입으로 저장하여 정렬 및 기간 조회 성능 저하 발생

- 원인
인덱싱 및 데이터 타입이 성능에 미치는 영향에 대한 이해 부족

- 해결
DATE 타입으로 변경하고 JOIN 구조 개선을 통해 쿼리 실행 속도 향상

- 배운 점
적절한 데이터 타입 선택과 쿼리 튜닝은 전체 시스템 성능에 직접적인 영향을 미침

## 8. 실행 환경 불일치 문제
- 문제
로컬 환경과 서버 환경의 Python 버전 차이로 인한 특정 라이브러리 실행 오류

- 원인
프로젝트 개발 환경 표준화 및 환경 관리 도구 활용 미흡

- 해결
Docker 또는 환경 설정 파일을 통해 서버 기준으로 실행 환경 표준화

- 배운 점
협업 및 배포 환경에서는 환경 격리와 버전 표준화가 필수적임을 확인

## 9. 대용량 데이터 처리 전략 부재
- 문제
전체 데이터를 한 번에 메모리에 로딩하여 시스템 다운 및 성능 저하 발생

- 원인
대규모 데이터셋 처리를 위한 분할 처리 로직 미비

- 해결
Chunk 단위 처리 또는 Generator 기반의 로직 도입으로 메모리 점유율 최적화

- 배운 점
데이터 규모에 따라 메모리 효율을 고려한 스트리밍 처리 전략이 필요함

## 10. 파이프라인 오류 전파
- 문제
특정 연도 데이터 누락이 전체 분석 결과의 신뢰도 하락으로 이어짐

- 원인
데이터 파이프라인 단계별 검증 및 로깅 프로세스 부재

- 해결
단계별 적재 완료 여부 확인 및 데이터 누락 방지를 위한 재적재 프로세스 추가

- 배운 점
파이프라인의 각 단계는 독립적인 검증과 오류 복구 메커니즘을 갖춰야 함

## 11. 문자열 데이터 불일치 이슈
- 문제
동일해 보이는 역명이 공백이나 접미어 차이로 인해 JOIN/집계 오류 발생

- 원인
시각적으로 확인하기 어려운 보이지 않는 문자나 불규칙한 표기 방식 존재

- 해결
HEX() 함수로 실제 값 확인 후 TRIM, REPLACE를 통해 데이터 정제

- 배운 점
문자열 데이터는 시각적 확인이 아닌 실제 바이너리 값 기반의 검증이 필요함

## 12. 컬럼명 특수문자 및 공백 오류
- 문제
특정 시간대 컬럼 조회 시 값이 존재함에도 모두 0 또는 NULL로 반환됨

- 원인
컬럼명 내 특수 공백 포함으로 인해 쿼리문 상의 명칭과 실제 메타데이터 불일치

- 해결
SHOW COLUMNS 및 LENGTH 확인을 통해 정확한 명칭으로 수정 및 사용

- 배운 점
외부 데이터 임포트 시에는 메타데이터 조회를 통한 컬럼명 검증이 선행되어야 함

## 13. 데이터 구조 통합 시 매핑 오류
- 문제
Raw 적재 단계에서 승차와 하차 값이 비정상적으로 섞여 들어가는 현상 발생

- 원인
UNION ALL 결합 과정에서 각 데이터 소스별 컬럼 순서 및 매핑 정의 미흡

- 해결
승하차 데이터를 명확히 구분하여 컬럼별로 할당되도록 쿼리 재설계 및 검증

- 배운 점
이종 데이터를 통합할 때는 각 소스별 데이터 매핑 정의서를 철저히 준수해야 함

## 14. 역명 표준화 실패로 인한 JOIN 오류
- 문제
Intermediate 테이블 생성 시 특정 역의 ID가 매핑되지 않고 NULL로 잔류

- 원인
데이터 소스별 역명 표기 방식("역" 유무, 괄호 포함 등)의 차이

- 해결
REPLACE 및 SUBSTRING_INDEX를 활용한 명칭 표준화 후 JOIN 수행

- 배운 점
문자열 기반 식별자 매핑 시에는 사전 전처리를 통한 데이터 정규화가 필수임

## 15. 데이터베이스 Collation 충돌
- 문제
테이블 간 JOIN 수행 시 Illegal mix of collations 오류 발생으로 쿼리 중단

- 원인
테이블별로 설정된 문자셋 정렬 규칙(Collation)이 상이하여 비교 연산 불가능

- 해결
JOIN 조건문에 COLLATE 명령어를 사용하여 정렬 규칙을 일시적으로 통일

- 배운 점
DB 설계 시 인코딩뿐만 아니라 정렬 규칙(Collation)의 일관성 유지가 매우 중요함

## 16. 1:N JOIN에 의한 데이터 증폭
- 문제
정제 과정 중 승하차 합계가 원본(Raw) 데이터 대비 비정상적으로 증가

- 원인
JOIN 대상인 마스터 테이블에 중복된 역명 정보가 존재하여 1:N 매핑 발생

- 해결
마스터 테이블을 유니크하게 재구성하는 서브쿼리 적용 후 매핑 수행

- 배운 점
JOIN 전 대상 테이블의 고유성(Uniqueness) 검증은 데이터 무결성 확보의 핵심임

## 17. 단계별 정합성 검증 프로세스 부재
- 문제
최종 마트 단계에서 데이터 이상을 발견하여 역추적 및 원인 파악에 과다한 시간 소요

- 원인
Raw-Int-Mart 계층별 데이터 정합성 검증 절차 누락

- 해결
Row Count 비교 및 수치 데이터 합계 검증 등 단계별 QA 로직 도입

- 배운 점
데이터 파이프라인의 완성도는 단순 적재가 아닌 단계별 검증 체계 구축에 있음

## 18. 데이터 계층화 설계 (Raw / Int / Mart)
- 문제
단일 테이블 기반 처리 시 비즈니스 로직이 복잡해지고 데이터 확장성 저하

- 원인
데이터 가공 단계별 역할 정의 및 레이어 분리 설계 부족

- 해결
Raw(보존), Int(표준화), Mart(집계)로 계층을 분리하여 파이프라인 구조화

- 배운 점
데이터 레이어 분리는 유지보수성을 향상시키고 데이터 신뢰성을 보장하는 핵심 구조임

## 19. Mart 테이블 재현 불가 이슈
- 문제
기존에 생성한 03_mart_daily_trend 테이블을 동일하게 재생성하지 못함

- 원인
테이블 생성 쿼리(DDL 및 집계 로직)를 별도로 저장하지 않아 정확한 로직 재현 불가. 또한 공휴일 JOIN, 날짜 변환, 문자열 처리 등 복합 로직으로 인해 미세한 차이 발생

- 해결
기존 테이블 구조 및 데이터를 기반으로 쿼리를 재작성하고 반복 검증 수행하였으나 완전히 동일한 결과 생성에는 실패

- 배운 점
집계 테이블 생성 쿼리는 반드시 버전 관리되어야 하며 데이터 결과보다 쿼리 자체의 재현 가능성이 더 중요함

## 20. 데이터 집계 기준 변경에 따른 이상치 발생
- 문제
2018년을 기점으로 전 역의 유동인구 데이터가 약 2배 증가하는 이상 현상 발생

- 원인
Row 수는 동일하나 개별 수치만 증가한 것을 확인하였으며, 공공데이터 소스의 집계 기준 변경으로 판단됨

- 해결
데이터 일관성 유지를 위해 2018년 이후 데이터에 가중치(0.5)를 적용하는 정규화 로직을 백엔드에 반영

- 배운 점
외부 데이터 활용 시 데이터 정의 및 집계 기준 변경 여부를 반드시 사전에 검증해야 함

## 21. Docker Compose 환경 설정 및 네트워크 오류
- 문제
Docker Compose 환경에서 API 호출 시 엔드포인트 중복 및 서비스 간 통신 오류 발생

- 원인
/api root path 설정과 라우터 엔드포인트 중복 정의
컨테이너 간 통신 시 localhost 기준 설정 사용
Python 버전 및 볼륨 마운트 설정 미흡
포트 설정을 서비스 내부와 게이트웨이에서 중복 적용

- 해결
API 경로 구조 재정의 및 라우터 중복 제거
서비스 간 호출 주소를 컨테이너 이름 기반으로 변경
Docker Compose 내 볼륨 및 Python 버전 설정 명확화
포트 설정을 게이트웨이 기준으로 일원화

- 배운 점
컨테이너 환경에서는 네트워크, 경로, 실행 환경이 로컬과 다르므로 설정 일관성이 중요함

## 22. 초기 로딩 성능 저하 및 UI 동기화 문제
- 문제
초기 화면 로딩 시 불필요한 컬럼 조회로 인해 응답 속도 저하 및 마커/목록 동기화 문제 발생

- 원인
전체 컬럼을 한 번에 조회하는 비효율적인 쿼리 구조
마커 선택 시 지도 중심 이동 및 데이터 기준 표시 로직 미흡

- 해결
서브쿼리를 활용하여 필요한 컬럼만 선조회하도록 쿼리 최적화
마커 선택 시 지도 중심 이동 및 목록과의 동기화 로직 구현
데이터 기준일을 연/월 단위로 표시하도록 UI 개선

- 배운 점
성능 최적화는 데이터 조회 단계에서부터 설계되어야 하며, 사용자 인터페이스와 데이터 로직의 일관성이 중요함
